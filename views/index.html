Analysis

The data used to create this model came from the IEEE-CIS Fraud Detection competition on Kaggle.com


Accuracy

After data cleaning, feature engineering, and several iterations of training, this model achieved a 0.9604 ROC-AUC score on a 33% holdout dataset. We chose this metric because the competition evaluated models based off AUC scores, allowing us additional data to test on. When tested against this larger holdout set provided by Kaggle, the model achieved an accuracy of 0.913. This accuracy score has the potential to improve with further development, feature engineering, and more time/computational resources to train. We believe this model has a high potential for practical applications, given that innovation in this area would lead to reducing fraudulent activity and improving customer experience.

Features

Engineered Feature #1

The possibilities for feature engineering in this dataset were numerous, however hindered by the lack of column labels(due to privacy agreements with the data source's customers). We deduced that TransactionDT is a time variable measured in minutes(86400 is the number of minutes in a day). From this variable, we engineered two additional features: hour of day and day of week. These two features seemed potentially useful in the case that fraudulent activity had some correlation with or clustered around these variables. The labels 'hour' and 'day' in the feature importance graph above are indicative of these features. As you can see, both made meaningful contributions to the model(totaling a .009 ROC-AUC score increase).

Engineered Feature #2

Our second engineered feature was facilitated by a question on the Kaggle message board asking about cases where the transaction amount contained three decimal places. We decided that the prescence of more than 2 decimal places was indicative of a foreign transaction being multiplied by an currency exchange rate, effectively giving us a feature detecting use of another currency. Because a large amount of column names are missing, there is no way to determine if some feature already exists in the data set which indicates foreign activity. However, we found this feature to be highly independent of other variables and a valuable addition to the dataset with an ROC-AUC increase of 0.0067.

Next Steps

As we attempt to bring this product to market, the clear first step is to continue improving our model. With a ROC-AUC score of 0.9604, we've made a great start, however there are many more ways we can boost the fidelity of our model. We will continue to engineer new features, as the two we had time to attempt made valuable contributions to our model. We will also attempt to locate relevant outside data sources to implement and interpret existing data. For example, one existing column is device type; converting this column to display operating system may yield interesting results. The last step we will take to improve our predictions is blending multiple uniquely-optimized models together for marginal gains over each other. We believe with more time we can notably improve the predictions of this already significant model.

The next step on the road to marketability is to interact with Vesta, the corporation which hosted the Kaggle data challenge. The most direct route to market would be to partner with Vesta to help them predict transaction fraud. This, of course, assumes that we can improve our models to the highest accuracy they have seen. Even if this is unlikely, it is worth pursuing if simply to gain insight into the data Vesta collects. If we can connect our predictive features with data collected on Vesta, we can continue to bring this product to market in other ways. Namely, we can create valuable insights for other banking/transaction-based companies to implement in their own detection.

The final step to bring this product to market is to pitch our improved model to various banking/payment companies. If we are able to determine the relevant data which correlate to our features or implement a new model on a company's data, we have a simple path to pitching our product. It is also possible that we can use our model to visualize localaties in which fraud occurs frequently and analyze the confluence of variables. There are several potential ways to implement this product in relevant businesses and we look forward to exploring those options.
